Location Binding Support

Why Location Binding [Index]

 

This design captures the support for the location binding such that the uri locations can be extracted in a binding file rather than storing in topology. Currently, the file URI's are explicitly stored in the topology file. This limits the portability of the topology file. Additionally, in some cases there is no way to get to the source information from the TDM model object. The location binding support will provide the means to capture the source linkage in first class manner rather than hiding it in annotations.

 

Location Binding Overview [Top]

 

The location binding file will store the explicit URI locations for any DeployModel object(DMO). This will provide for the source linkage of the DMO. Additionally, the binding entry in the topology resource will only contain the key's while the actual value will be stored in the new binding resource. This will provide for storing multiple bindings per topology based on the product context in use. File Artifact on the DMO will be used to capture this information.

 

This design will address the following key requirements location binding:

  • A new location binding schema
  • Location Binding Service to resolve or encode the entries
  • File Artifact - File URI resolution
  • Preference setting For Binding to use
  • Updates needed for Providers to Create the File Artifact
  • Resource Management Changes for Location Binding Resource
  • Updates to Property Sheet to use the resolution API for file URI addition/display
  • Export Changes
  • Location Binding Editor - Post M5

 

Note: The concept document for this is: https://radical.raleigh.ibm.com/rad/34249.html

 

Preference Setting [Top]

 

The location binding support can optionally be turned off. We will be having a preference setting in the tool to indicate the environment context in use and also whether to use the location binding resource or inline the information with-in the topology resource. There needs to be a mechanism to register product's environment context and will be captured as another design in M6.

 

In M5, we will add a preference in the Topology Editor preferences to specify whether to use the location binding file or not. This should be a product preference in future if we need to.

Location Binding Schema [Top]

 

There is a requirement to store multiple binding for a topology per product context.

.

<locationBinding topology="namespaveA.Top1" name="name1" context="rdm" >
 
  <bindingEntry dmoPath="populate.populate" location="file://d:/Sample/populate.ddl"/>
  <bindingEntry dmoPath="WebComponent" location="platform://WebComponent"/>  

  <bindingEntry dmoPath="WebComponentA" location="cvs:Server@2403..."/> 

</locationBinding>

The location binding resource will be suffixed with <locName>.locbinding and stored along with the topology resource. The locName is derived from the topologyname.

We can have more than one binding file per product context. Only one file will be active for use. If there are multiple binding files for a product context, then user will have to indicate which file to use. This information will be stored in the user preference store.

There will be further refinement of using the location binding file based on the context attribute. This will provide an hint to indicate which location file to use when multiple are present with different product context.

Multiple binding file support is currently targetted to M6.

File Artifact Updates [Top]

 



The getFileURI method needs to be updated such that it invokes the locationBindingservice to resolve the binding information based on the location file in use. If there is no key present for the fileartifact in the binding file, then an entry is made in the binding file using the resolution service. When an new fileURI is added, it should be always done through the LocationBindingService such that key is added to the binding file and the topology fileURI is not used. This is done by making a check at the persistence time for dmoPath for all the entries in the location binding file before serializing out. When the topology is saved, all stale entries from the binding file should be removed.
