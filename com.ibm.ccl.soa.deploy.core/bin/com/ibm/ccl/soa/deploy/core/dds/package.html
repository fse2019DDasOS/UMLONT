The SOA Deploy Deployment Topology decorator semantic of the SOA Deployment Tooling allows developer to deploy the application artifacts on to the infrastructure.

SOA Deploy Deployment Topology Decorator Semantic

SOA Deployment Business Application Topology Domain [Index]

 

The SOA Deployment Deployment Topology is an extension domain of the SOA Deployment Editor which will allow Deployment Architects to deploy their business components onto the infrastructure elements.. This document will focus on the implementation of the decorator semantic that will customize the user experience for the deployer usage. The types and the visual artifacts will be filtered and/or added, based on the needs of the deployment topology developer.

 

Deployment Decorator Semantic (DDS) Overview [Top]

 

The DDS is aimed at easing the user experience for the deployment topology development. The core decorator semantic framework is used to provide an extension of deployment decorator semantic. The domain types and the UI views, pallette entries are filtered along by providing simple contributions. Deployment Topology allows operations integrator/developer to prepare and publish the units in the topology. This can either be used during development to unit test the application or can be used to publish contents of the topology to operations.

 

Simple Usage Scenario

  • A business topology with a Web component and a infrstructure topology with a Tomcat 5.0
    server unit associated with a runtime id is available.
  • Create a new topology and set the type as 'Deployment'
  • Drag and drop a business application topology from the navigator onto this topology.
  • Drag and drop the infrastructure topology from the navigator onto this topology.
  • Host the Web unit on the Tomcat Unit using the Hosting Link.
  • Save the topology.
  • Right click and Publish the topology. As a result the web component should be deployed on the Tomcat.

 

 

This design will address the following key requirements of a business application decorator semantic contribution:

  • Extension to contribute a DDS
  • EMF Type enablement extension for DDS
  • UI Type enablement extension for DDS
    • Enabled Views/Menus/Pallette for DDS and their content
  • Extended validation for DDS

 

Note: The decorator demantic design  is used as a reference (com.ibm.ccl.soa.deploy.core\designs\com\ibm\ccl\soa\deploy\core\decoratorsemantic\package.html)

 

Plugin Contribution[Top]

 

The DDS is a shared decorator semantic between the RDM and TDM tools. The base definition will be defined in the core plug-in. All the extension plug-ins will contribute the pattern bindnings to this definition.

We ruled out the separate plug-in for the DDS as it will make it very lightweight plug-in.

 

DDS Extension Contribution[Top]

 

The following basic extension needs to be contributed in the core plug-in such as to define the DDS.

<!-- Deployment Topology Decorator Semantic -->
<extension
id="com.ibm.ccl.soa.deploy.dds"
name="com.ibm.ccl.soa.deploy.dds"
point="com.ibm.ccl.soa.deploy.core.topology">

<decoratorSemantic

description="Deployment Topology is used by the architect/developer to define the hosting of the business component(s) or the realization of the conceptual nodes, from the business application topology(s) to the middleware servers defined in the infrastructure topology(s). Use this topology type, you can publish the business component(s) to the middleware these are hosted on in the infrastructure topology. The use of this topology can either be done for targeting the unit test or the production environment."


id="com.ibm.ccl.soa.deploy.dds"

icon="icons/dds.gif"

name="Deployment"/>

</extension>

<!-- *************************************************************** -->

 

 

 

 

Core Pattern Binding Contribution[Top]

The following will be contributed by the core plug-in as a pattern binding.

 

 

<!-- Deployment Topology Decorator Semantic -->

<decoratorSemanticPatternBinding decoratorSemanticId="com.ibm.ccl.soa.deploy.ds">
<typeContent>
<include pattern="*"/>
</typeContent>
<unitProviderContent>

<include pattern="*"/>
</unitProviderContent>
<validatorContent>
<include pattern="*"/>
</validatorContent>
</decoratorSemanticPatternBinding>

Validation Enablement[Top]

 Each of the domain plug-ins will contribute a pattern binding for their respective validators if there is a need to exclude any. Since we are not restricting any content, no suitable content found to be excluded.
