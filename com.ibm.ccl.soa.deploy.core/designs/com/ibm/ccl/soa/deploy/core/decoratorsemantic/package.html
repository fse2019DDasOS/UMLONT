The Decorator Semantic extension mechanism allows contributors to the SOA Deploy Tooling the ability to declare a semantic to enable specific object types and UI artifacts that are relevant to the desired tooling mode.

SOA Deploy Decorator Semantic

 

Abstract [Index]

 

This document describes the design and usage of a "Decorator Semantic" notation on a topology in the SOA Deploy tooling. The decorator semantic can be used to refine the contents of the editor experience for a user of the tooling, particular to the mode they are working in, or the type of topology that are currently editing.

Additional design documents may be required to fully define the specific consumptions of this mechanism, such as the topology creation editor, palette loading, etc.

The document will focus on three primary areas of design:

  • Required model changes
  • The extension mechanism to declare a new decoratorSemantic
  • The extension mechanism to define content to a decoratorSemantic

Bugzilla item for this review: 11501

Web link: https://cs.opensource.ibm.com/scm/viewvc.php/*checkout*/development/designs/11501_package.html?root=soa-deploy-tool

 

Core Model Changes[Top]

 

The changes required to the core models to support a decoratorSemantic are to simply add an attribute to the topology element called "decoratorSemantic". This attribute will be of type xsd:String, optional, and will be single valued.

The proposed change to the core.xsd schema is highlighted below:


<!-- Topology -->
<element name="topology" type="core:Topology"/>
  <complexType name="Topology">
  <complexContent>
    <extension base="core:DeployModelObject">
      <sequence>
        <element minOccurs="0" maxOccurs="unbounded" ref="core:unit" ecore:name="units">
        </element>
      </sequence>
      <attribute name="decoratorSemantic" type="string" use="optional">
    </extension>
  </complexContent>
</complexType>

It will be the responsibility of topology-creation mechanisms, such as the "New Topology Wizard", to set the value of this attribute.

Comments/Questions/Issues:

  1. Should attribute be required? Assumption is that if the decoratorSemantic is not defined, all available content is enabled for the topology.
  2. Originally, a multi-valued attribute was considered, allowing multiple semantics to be defined on a topology. This concept was backed out, since it may cause confusion in cases where there is overlap between semantics. For example, one semantic enables tomcat content, while another one specifically disables it. If both are set, which one wins? For M3, it is proposed that only a single semantic can be defined per topology. In this case, the topology wizard will only support a dropdown of available decorator semantics, and not a set of checkboxes.
  3. We should be able to create a topology that reuses existing decorator semantics, which is why we were considering multi-valued attributes. Could also be achieved by allowing the user to extend or aggregate existing decorator semantics to define a new semantic. This still leads to the same issues of overlap and confusion when items conflict. Some potentially complicated priority rules would need to be spelled out for this to be fully functional

Post M3 Items to consider:

  • Multi-valued attribute
  • Prioritization rules
  • Ability to extend or aggregate existing decorator semantics into a new decorator semantic.

 

 

Decorator Semantic contribution [Top]

 

This section describes the proposed extension mechanism to declare a decorator semantic to the tooling.

The approach used here is similar to that used by the FunctionGroup mechanism that was created for potential inclusion in WTP. Concepts from that solution have been brought forward into this mechanism, as it is in many ways doing the same thing. The overview is that there are two separate extension mechanisms that are defined :

1. The decoratorSemantic definition itself, with a unique identifier.
2. The pattern bindings for the decorator semantic, which specifically refer to a decorator semantic (by id) that the declared content should be enabled for.

Extension Point Modifications

The proposal here is to add an element to the exiting Topology extension point. This element would contain a name attribute, an id attribute, which is the string key for the new decorator semantic, an icon attribute, and a description attribute.

The minimum occurrences of this element will be 0.
The maximum occurrences of this element will be unbounded.

Extension Point Details

This element will have the below structure in the extension point .exsd schema

   <element name="decoratorSemantic">

      <complexType>

         <attribute name="id" type="string" use="required"/>

         <attribute name="name" type="string" use="required"/>

         <attribute name="description" type="string"/>

         <attribute name="icon" type="string"/>

      </complexType>

    </element>

id : The unique identifier for this decorator semantic (required)
name : The display name of this decorator semantic (required)
description : The description of this decorator semantic (optional)
icon : Location of an image file for this decorator semantic (optional)

Example

      <decoratorSemantic

            description="A Test Decorator Semantic"

            id="test.testDecoratorSemantic"

            icon="images/someicon.gif"

            name="Test Decorator Semantic"/>

Comments/Questions/Issues:

1.                   Assuming for M3 that the topology extension point is the best place for this contribution. As opposed to the domain level or perhaps even a standalone extension point. Domain extension may not be the best place since a semantic can potentially cross domains.

 

Decorator Semantic Pattern Binding contribution[Top]

 

This section of the design details how to declare content in the tooling based on a specific decorator semantic.  This will be achieved by defining a decoratorSemanticPatternBinding element that is used to declare the enabled content for a specific decoratorSemantic.  This content is divided into several categories (menus, validators, types, etc.), and can be included or excluded as “patterns”, following java.util.regex syntax, bound to a decorator semantic.

By default, a topology without a decorator semantic defined will contain all content.

Extension Point Modifications

The proposal here is to add an element to the exiting Topology extension point. This element would contain an attribute to associate the bindings with a specific decorator semantic ID, and a subelements to define the different types of pattern bindings.

The minimum occurrences of this element will be 0.
The maximum occurrences of this element will be unbounded.

The content subelements are currently used to distinguish the different types of content that can be bound.  There are currently two such content elements defined: <typeContent> and <uiContent>.  Each content subelement can contain multiple <include> elements to define patterns for enablement.

   <element name="decoratorSemanticPatternBinding">

      <complexType>

         <sequence>

            <element ref="typeContent" minOccurs="0" maxOccurs="1"/>

            <element ref="uiContent" minOccurs="0" maxOccurs="1"/>

         </sequence>

         <attribute name="decoratorSemanticId" type="string" use="required">

         </attribute>

      </complexType>

   </element>

   <element name="typeContent">

      <complexType>

         <sequence>

            <element ref="include" minOccurs="1" maxOccurs="unbounded"/>

            <element ref="exclude" minOccurs="0" maxOccurs="unbounded"/>

         </sequence>

      </complexType>

   </element>

 

   <element name="menuContent">

      <complexType>

         <sequence>

            <element ref="include" minOccurs="1" maxOccurs="unbounded"/>

            <element ref="exclude" minOccurs="0" maxOccurs="unbounded"/>

         </sequence>

      </complexType>

   </element>

 

   <element name="validatorContent">

      <complexType>

         <sequence>

            <element ref="include" minOccurs="1" maxOccurs="unbounded"/>

            <element ref="exclude" minOccurs="0" maxOccurs="unbounded"/>

         </sequence>

      </complexType>

   </element>

 

   <element name="viewContent">

      <complexType>

         <sequence>

            <element ref="include" minOccurs="1" maxOccurs="unbounded"/>

            <element ref="exclude" minOccurs="0" maxOccurs="unbounded"/>

         </sequence>

      </complexType>

   </element>

 

   <element name="unitProviderContent">

      <complexType>

         <sequence>

            <element ref="include" minOccurs="1" maxOccurs="unbounded"/>

            <element ref="exclude" minOccurs="0" maxOccurs="unbounded"/>

         </sequence>

      </complexType>

   </element>

 

   <element name="paletteEntryContent">

      <complexType>

         <sequence>

            <element ref="include" minOccurs="1" maxOccurs="unbounded"/>

            <element ref="exclude" minOccurs="0" maxOccurs="unbounded"/>

         </sequence>

      </complexType>

   </element>

 

   <element name="propertySheetContent">

      <complexType>

         <sequence>

            <element ref="include" minOccurs="1" maxOccurs="unbounded"/>

            <element ref="exclude" minOccurs="0" maxOccurs="unbounded"/>

         </sequence>

      </complexType>

   </element>

 

   <element name="userDefinedContent">

      <complexType>

         <sequence>

            <element ref="include" minOccurs="1" maxOccurs="unbounded"/>

            <element ref="exclude" minOccurs="0" maxOccurs="unbounded"/>

         </sequence>

         <attribute name="name" type="string" use="required">

      </complexType>

   </element>

   <element name="include">

      <complexType>

         <sequence>

            <element ref="enablement" minOccurs="0" maxOccurs="1"/>

         </sequence>

         <attribute name="pattern" type="string" use="required"/>

      </complexType>

   </element>

   <element name="exclude">

      <complexType>

         <sequence>

            <element ref="enablement" minOccurs="0" maxOccurs="1"/>

         </sequence>

         <attribute name="pattern" type="string" use="required"/>

      </complexType>

   </element>

   <element name="enablement">

      <complexType>

         <sequence minOccurs="0" maxOccurs="unbounded">

            <element ref="not"/>

            <element ref="and"/>

            <element ref="or"/>

            <element ref="instanceof"/>

            <element ref="test"/>

            <element ref="systemTest"/>

            <element ref="equals"/>

            <element ref="count"/>

            <element ref="with"/>

            <element ref="resolve"/>

            <element ref="adapt"/>

            <element ref="iterate"/>

         </sequence>

      </complexType>

   </element>

Example

      <decoratorSemanticPatternBinding decoratorSemanticId="test.testDecoratorSemantic">

         <typeContent>

            <include pattern="com.ibm.some.object.types.*"/>

         </typeContent>

         <viewContent>

            <include pattern="com.some.ui.views.*"/>

         </viewContent>

      </decoratorSemanticPatternBinding>

Pattern Binding Details

Each pattern binding contains multiple categories of content, each of which can contain patterns of content to include or exclude in the semantic.  Additionally, each include or exclude pattern can optionally use Eclipse enablement to add additional conditions around the inclusion of a pattern.  In these cases, both the enablement condition and the pattern expression must be satisfied for the pattern to be included (or excluded).

The following are the supported categories of content for the pattern binding:

·         Type content

·         Menu content

·         Validator content

·         View content

·         Property sheet content

·         Palette entry content

·         Unit provider content

·         User-defined content

The below section describes each of these categories, and their usages, in more detail.

Type Content : <typeContent>

This category allows the semantic to specifically enable object types for this semantic.  The required syntax for the pattern expression is the java namespace of the class.

The usage of target namespace is beneficial to prevent the same type being defined in multiple places (i.e. palette, context menu, etc.). However, a downside to this is that it limits the refinement to what object types have been declared in the model. For example, today there are not separate unit types for WAS6, WAS51, etc. They share the same type. This ultimately is caused by our lack of a formal resource type system in the tooling, and each contributor is left to define and map types independently.

The primary difference between the typeContent category and the others, is that the APIs to determine type enablement will take an object argument, as opposed to a string argument.

Example

      <decoratorSemanticPatternBinding decoratorSemanticId="test.testDecoratorSemantic">

         <typeContent>

            <include pattern="com.ibm.some.object.types.*"/>

            <exclude pattern="com.ibm.some.object.types.not.this.one"/>

 

         </typeContent>

      </decoratorSemanticPatternBinding>

Menu Content : <menuContent>

This category declares menus that should be enabled in the decorator semantic, based on their respective menu IDs.  Currently (M3), this mechanism is queried to load the context menus for the deployment diagram editor.

Validator Content : <validatorContent>

This category declares the set of domain validators that should be enabled for this decorator semantic.  The validators are specified by their classname.  This allows the semantic provider the ability to disable unwanted domain validators in a topology with a specific semantic.

API has been added to the DomainManager to query decorator semantic ID when loading domain validators.

View Content : <viewContent>

This category allows the ability to include or exclude particular views from the deployment perspective in the given decorator semantic.  The views should be identified by their Eclipse ID.

Property Sheet Content : <propertySheetContent>

This category allows the ability to include or exclude particular property sheets from the deployment perspective in the given decorator semantic.  The property sheets should be identified by their Eclipse ID.

Palette Entry Content : <paletteEntryContent>

This category allows the user to customize the set of available entries in the deploy editor palette.  Entries can be included or excluded as required.  The paletteEntries should be declared by their GMF palette ID.

Post M3: Find a way to automatically associate paletteEntryContent with typeContent.  This may be expensive since the GMF paletteEntry definition and the associated EObject (from which type could be derived) are defined in separate extension points.

Unit Provider Content : <unitProviderContent>

This category allows the user to declare which unit providers should be enabled for the given decorator semantic.  The providers should be identified by their classname.

APIs have been added to the UnitLifeCycleManager to allow lookup of providers based on input/output object and a decorator semantic ID.

User Defined Content : <userDefinedContent>

This is a special category to allow users to dynamically group semantic content bindings in their own category, other than those previously mentioned.  The functionality is similar to the previous categories, with an extra ‘name’ attribute required to define the category name for these bindings.  All bindings for the same category can be accessed through API in the DecoratorSemanticService that accepts both a semantic ID and a category name.

Pattern framework
The above categories will all use a pattern-matching framework to allow ease and flexibility with declaring enabled content. This can utilize the approach used by Eclipse capabilities. The following concepts should be supported:

  • Specific enablement of exact types
  • Enablement of a group of types via wildcarding
  • Exclusions of types from a specific pattern.


Initially, the support will be based on that which is supported by the java.util.regex framework.   A string will be compared against a specified pattern to determine if a match is found or not.  Additional complexity with matching rules can be implemented in later milestones.


The java.util.regex framework syntax is not always the same as what is typically used in simple wildcarding schemes.  For example, to include all content, one must include ".*" instead of "*" (note the preceding '.'). 


The logic used in the implementation will be to give <exclude> patterns priority over <include> patterns.  For instance, if a given pattern is found in the exclude list, it is deemed to be excluded from the semantic content, regardless if it is listed in the include list or not.

 

Post M3 Items to Consider

  • Enablement on type content entries, to allow for cases when the object namespace is not enough information to fully restrict the content.  For example, the ability to enable WAS6 but not WAS51, since there are not separate types for these elements.
  • More robust pattern matching support (exclusion) and priority rules to handle collisions.  For example, semantic one enables a type, while semantic two disables that same type.  Which has priority?
  • Enhanced type content support for validators; allow the ability to replace or disable the default validator for a given type, or declare a new validator inline.  This will likely heavily rely upon a better validation framework that supports more fine-grained validators beyond the existing domain level.

 

 

API Support [Top]

 

The following APIs will be made available for accessing decorator semantic information. Assuming the proposed solution, these APIs will be accessible from the TopologyManager or the DecoratorSemanticService.

public String[] getAllDecoratorSemanticNames();

public String getDecoratorSemanticDescription( String semanticName );

public String getDecoratorSemanticId( String semanticName );

public boolean isDmoMemberOf( String decoratorSemanticId, DeployModelObject dmo );

public boolean isMenuMemberOf( String decoratorSemanticId, String menuId );

public boolean isPaletteEntryMemberOf( String decoratorSemanticId, String paletteId );

public boolean isPropertySheetMemberOf( String decoratorSemanticId, String propertyId);

public boolean isUnitProviderMemberOf( String decoratorSemanticId, String providerId );

public boolean isUserContentMemberOf(String decoratorSemanticId, String userCategory, String contentItem);

public boolean isValidatorMemberOf( String decoratorSemanticId, String validatorId );

public boolean isViewMemberOf( String decoratorSemanticId, String viewId );

public DecoratorSemanticDescriptor[] findAvailableDecoratorSemantics();

public DecoratorSemanticDescriptor findDecoratorSemantic(String id);

public DecoratorSemanticPatternBindingDescriptor[] findAvailableDecoratorSemanticPatternBindings();

public DecoratorSemanticPatternBindingDescriptor findDecoratorSemanticPatternBinding(String id);

  1. Any other APIs that may be required in M3?
  2. API names are certainly welcome for input.

 

Testing Requrements [Top]

 

The decorator semantic extensions and API will be verified by junits that test the follow:

  • Load and verify the declared decorator semantics in the workspace
  • Create a topology with a given decorator semantic
  • Verify enabled and disabled content for the given topology.
  • Test all categories, including userDefinedContent.

See com.ibm.ccl.soa.deploy.tests.extension.DecoratorSemanticTests
