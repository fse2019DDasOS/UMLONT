The IBM Deployment Modeling Platform provides models and tooling elements to construct validatable metamodels of the Topology Definition Model (TDM).

IBM Deployment Modeling Platform Core Model Implementation Documentation

 

Abstract [Index]

 

The EMF-generated implementation classes of the IBM Deployment Modeling Platform core model include non-generated methods.  This document outlines the customizations that have been performed.

 

Core Model: Link [Top]

 

Deploy model links endpoints are maintained in-memory as EMF references, and are serialized as string URIs.  The deserialization of link references is performed lazily, when the EMF reference accessor method is invoked.

The DeployModelObject type, a common super-type of all deploy objects, defines a method for obtaining a fully qualified, or relative path to the object.  A path can be resolved in the context of another DeployModelObject.  The relevant methods are: getFullPath(), getRelativePath(DeployModelObject), resolve(String).

The abstract DeployLink type provides accessors for the string link endpoint URIs: getSourceURI() and getTargetURI().  Subtypes must implement the EMF reference accessors and setters, based on the expected type of the endpoints.  The DeployLinkImpl class provides transient attributes for catching the EMF reference.  It also provides helper methods getTargetObject(), setTargetObject(DeployModelObject), getSourceObject() and setSourceObject (DeployModelObject).  Subtypes can implement get/set source/target just delegating to these helper methods and casting the return value  For example, DeployLinkImpl includes:

       public Capability getTarget() {

            return (Capability) super.getTargetObject();

      }

      public void setTarget(Capability service) {

            super.setTargetObject(service);

      }

 

Core Model: Requirement.dmoEType [Top]

 

The value of the persisted Requirement.dmoType attribute is a string reference to the XSD subtype of DeployModelObject.  In memory, users get and set the EMF EClass object using the manually coded getDmoEType() and setDmoEType(EClass) methods.  For example, the EClass OperatingSystem would be serialized as “os:operatingSystem”.

 

During serialization, it is required that the XML namespace prefix of the all the types referred to in Requirement.dmoType be included in the root of the document.  Because it is possible that no instances of a namespace are instantiated in the topology, we must take extra steps to insure the consistency of the documents prefix.

 

In order to assure correct serialization of the XMLNS prefixes, the DeployEncryptingXmlHelper class overrides the packages() method to also include the XMLNS prefixes for Requirement#getDmoEType() types contained in the topology.
