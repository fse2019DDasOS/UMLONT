Abstract [Index]

The Java EE Domain defines the concepts and abstractions Java Enterprise Edition (EE) applications and their containers.

Java EE Components [Top]

Java EE Components represent the standard application components supported by Java EE, including Enterprise Applications, Application Clients, Enterprise Java Bean Modules, Web modules, and Java Connection Architecture modules. Each Java EE Component specializes the Business Application Domain's SoftwareComponent.

Each Java EE Component describes technology specific Services, and declares References which may or may not be technology specific.
Service Logic Agreements [Top]

For M3, two basic components will be defined that describe EJB modules and Web modules.

Each of these Java EE Components declares Services that are specific to its technology, such as service.ejb for EJBComponent or service.url for WebComponent.

Bundle Dependency Contracts [Top]

Each Java EE Component describes a bundle of code as an isolateable unit that can be built, included, or referenced in a build process. Each component exposes a Bundle Capability which can be referenced to express packaging and compilation dependencies. These Bundle Capabilities are linked with DependencyLinks, and do not express a decision about whether the two Components will be co-located, packaged together or one enclosed in the other. A separate build model must make decisions regarding build order and packaging for deployment.

For WebModule, the existing J2EEWebModuleCapability will satisfy the purpose of the more abstract BundleCapability. If a common base type is added, then J2EEWebModuleCapability will extend BundleCapability.

Testing Requrements [Top]

The Java EE Domain API will be tested by verifying the following:

  • A Service and Reference match if the Reference defines a compatible Interface with the Service.
  • A SoftwareComponent should be accessible from the palette or for any Java EE Module.
