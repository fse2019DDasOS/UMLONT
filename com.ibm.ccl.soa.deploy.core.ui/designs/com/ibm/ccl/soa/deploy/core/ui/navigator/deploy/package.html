Deployment Navigator Overview Documentation

Abstract [Index]

The deployment perspective currently has its own separate deployment navigator in our framework. The original purpose behind this architecture was to help customers conceptualize the ideas behind the deployment perspective. Namely, it would group projects inside of top-level virtual nodes so that it was obvious which projects were deployable/had topologies/etc.

This implementation is no longer practical, being that we are beginning to re-invent the Project Explorer. Most features offered by the project explorer are becoming needed in the deployment perspective, and it no longer makes sense to have a separate navigator. Many bugs that have arisen are a result of using the deployment navigator and would be resolved by switching to the use of the project explorer.

In the following initial design, we will discuss the replacement our current navigator with the Project Explorer used by other Eclipse perspectives and address some new functionalities to be implemented for the navigator in this milestone.

Bugzilla item for this review: 12204

NamespaceElementContentProvider:

Provides content for the deployment navigator tree in the form of NamespaceFragmentRoot and NamespaceFragment objects. Implements the IPipelinedTreeContentProvider.

An important distinction to make about this content provider is that it displays the NamespaceFragments as a flat structure rather than a heiarchal structure. NamespaceFragments are displayed a children of their NamespaceFragmentRoot and not as children of the NamespaceFragment that is their parent.

NamespaceElementLabelProvider:

Provides the label information for the deployment navigator tree for the NamespaceFragmentRoot and NamespaceFragment objects appearing in the tree.

This label provider displays NamespaceFragments with their fully-qualified names and not just the last segment of their fully-qualified name. This is necessary since the content provider for NamespaceFragments utilizes a flat structure where all NamespaceFragments that belong to a single NamespaceFragmentRoot are displayed a children of the same NamespaceFragmentRoot object. So fully-qualified names on the NamespaceFragment children are necessary in order to distinguish the "heirarchy" of the NamespaceFragments from other NamespaceFragments in the same NamespaceFragmentRoot.

EmptyNamespaceFragmentPropertyTester:

Is used within the defininition of a common filter for the deployment navigator to test if the current NamespaceFragment in the tree is "empty" or not.

Open Topology Dialog [Top]
Items:
  1. Add an Open Topology action to the Navigator

Details:
The latest concern is giving the user easy access to his/her topology files. The current suggestion is to use a open topology dialog similar to the "open resource dialog"/"jdt open type dialog". This dialog would be bound to a certain keyset in the deployment perspective (need to pick a keyset), and would also be available inside the navigation menu and also the action toolbar.

This would perform the same way the open resource dialog does today, but it would filter all resources except for .topology files. Rather than display content using a selection dialog (with file containers displayed underneath the dialog), we will use something similar to the project explorer's tree navigator. This way, we can use filtering on a tree-like structure in order to better convey the containment of similarly named items.

Proposed Changes:

  1. Extend SelectionDialog to build our own SWT Composite Dialog named OpenTopologyDialog

  2. Dialog content will be displayed by a FilteredTree

  3. Extend NamespaceElementContentProvider to return us the content we want for our Dialog

  4. Add this new action and command binding to the plugin.xml

Content Changes [Top]
Items:
  1. Add the ability to display imported Topologies

Details:
The Navigation will need to be able to now recognize imported topologies inside of a topology file. These will exist as URI references in the model. Imported topologies containing imports themselves will not be included when imported into another topology (i.e. the topology that is DnD onto the editor is the one that is imported; no consideration is made as to whether or not that topology has an import inside of it)

Imported Topologies should:

  1. Be displayed with other nodes underneath the topology
  2. Display with their own specific Icon and Text
  3. Not display their children (to avoid cluttering the navigation)
  4. When opened, be displayed in a seperate window, just like normal topologies
    5. -Navigator should jump to the actual location of the topology when opened

Proposed Changes:

  1. Add imports field to Topology

  2. Modify the TopologyItemProvider to return the imports in its list of children

    3. -This item is based on lineitem 12198
    -Should essentially get a list of imported topologies from the model, and if so then display them at the top of the children tree.

  3. Generate/Modify ImportItemProvider to display the imports

    4. -Need to modify to display imported topologies with their own label and icon
